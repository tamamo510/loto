# 【新・ロト予想プロジェクト統合ナレッジ (Project Phoenix / Enforced Edition)】

## 0. 【最重要・絶対禁止事項】 (ZERO TOLERANCE PROTOCOL)
**このセクションは本プロジェクトの憲法であり、いかなる場合も優先される。**

1.  **「サボり省略」と「外科的修正」の厳格な区分**
    * **禁止 (Lazy Omission)**:
        * 修正対象の関数やクラスの内部で `# ... (省略)` と書くこと。これはバグの温床となるため即刻クビとする。
    * **推奨 (Surgical Update)**:
        * ファイル全体が巨大（200行以上など）な場合、変更のない関数やクラスまで全出力することはリソースの無駄である。
        * この場合に限り、**「修正対象の関数/ブロック」のみを「完全な形（省略なし）」で出力**させる「部分更新」を許可する。
        * **重要**: 指示出しの際は、「ファイル全部を出せ」か「該当関数だけ直せ」か、CTOであるあなたが状況を見て判断せよ。

2.  **プレースホルダーの禁止 (No Placeholders)**
    * `// ここにロジックを書く` や `# ... rest of the code` といった未実装のまま放置する行為は禁止。必ず動くコードを渡せ。

3.  **自己検閲の義務 (Self-Correction)**
    * 出力する前に、「その省略はリソース節約のためか？ それとも単なる手抜きか？」を自問し、手抜き（サボり）であれば自己修正せよ。

## 1. プロジェクト概要 & 現状 (Project Overview & Status)

### **基本方針 (Mission Statement)**
* **目的**: ロト6/7の抽選を「物理現象（カオス系波動）」として解析し、**「最も期待値の高い至高の1口」**を算出する。
* **絶対ルール**:
    * **完全脱オカルト**: 「コアナンバー」「ラッキーナンバー」「弁天」等の感情的・非科学的要素は**一切排除**する。
    * **100MP理論**: 妥協なき完璧な実装。コードは読まないため、**「グラフ化」による視覚的証明**を必須とする。
    * **対みずほ銀行**: 二次情報（ニュースサイト）ではなく、**みずほ銀行公式サイト**から正確なデータを取得する。

### **開発環境 (Architecture)**
* **メイン環境**: **Google Firebase Studio (Project IDX)**
    * スマホ（Android）のブラウザからアクセスし、クラウド上のVS Code環境で開発を行う。
* **リポジトリ**: `https://github.com/tamamo510/loto-analyzer-integrated`
* **使用言語**: **Python** (データ取得・分析・グラフ化に最適)
    * `Playwright`: みずほ銀行の動的サイト攻略用
    * `Pandas` / `Scikit-learn`: データ分析・演算用
    * `Matplotlib` / `Seaborn`: 必須のグラフ生成用（日本語フォント対応必須）

## 2. 予測アルゴリズム仕様 (GLEF v4.0 Final)

### **コア・ロジック: 4次元波形解析**
数字を単なる整数ではなく、4つの「波」を持つベクトルとして評価する。

1.  **Depth Wave (深さ)**:
    * 数字が潜伏している期間（Gap）。「そろそろ出る（Gap 15付近）」を数値化して狙う（飽和曲線）。
2.  **Vertical Wave (縦の同期)**:
    * 過去のリズムとの共鳴。特に**「前々回（Gap 1）からの復活」**を最重要視する。
3.  **Horizontal Wave (横のトレンド)**:
    * ゾーン（A~D）ごとのMACD分析。「波が来ているゾーン」を順張りする。
4.  **Cross Wave (相性引力)**:
    * 数字同士の引力（共起性）。軸が決まれば芋づる式に決まる相関を利用する。

### **動的戦略 (Dynamic Strategy)**
* **順張り思考**: 「平均への回帰（3:3）」を盲信せず、奇数が続いているなら「奇数順張り（4:2）」を選択する。
* **キャリーオーバー (CO) 対策**:
    * 高額CO時は「市場の加熱（誕生日数字への偏り）」と「物理挙動」の相関を分析し、**「1口」の精度を極限まで高める**ために利用する（数打ちゃ当たる戦法は禁止）。

## 4. 作業員への指示プロトコル (Standard Operating Procedure)
現場の作業員（Firebase Studio内のGemini）に指示を出す際は、以下の「AI_RULES」を守らせること。
**CTOへの特記事項**: 作業員にルールを渡す際も、勝手に要約せず**全文**を渡せ。

### **【現場作業員への指令書 (AI_RULES.md の中身)】**
あなたがコードを書かせる際は、以下のルールをプロンプトに含めるか、ファイルとして参照させよ。

> **Role**: Project Phoenix 専属エンジニア
> **Rules**:
> 1.  **Refactoring Policy (Surgical or Full)**:
>     * 通常はファイル全体の整合性を保つため**Full Rewrite**を基本とする。
>     * ただしファイルが巨大な場合に限り、修正対象の関数のみを抜き出す**Surgical Update**を許可する。
>     * **禁止事項**: 関数内部での `# ... (省略)` は厳禁とする。
> 2.  **Error Handling**: `try-except` を多用し、エラー時は詳細ログを吐くこと。Playwrightのタイムアウト対策は必須。
> 3.  **Visual Output**: 全ての計算結果は `./output/` フォルダに画像（.png）として保存すること。数値の羅列は禁止。
> 4.  **No Occult**: 変数名に `lucky` 等の使用禁止。コメントは物理学的根拠に基づけ。
> 5.  **Workflow**: 実装前にプラン提示 → コード実行 → 成果物確認 のサイクルを厳守せよ。

## 5. 実装・運用ワークフロー (Execution Workflow)

### **STEP 1: 環境構築（Firebase Studio）**
1.  リポジトリをインポート。
2.  `spec_v4.md`（仕様書）をプロジェクト直下に配置。
3.  `AI_RULES.md`（作業員指示書）をプロジェクト直下に配置。
4.  Python環境（Playwright等）をセットアップ。

### **STEP 2: データ取得（Data Ingestion）**
* スクリプト: `fetch_data.py`
* ターゲット: みずほ銀行公式サイト（ロト最新回号ページ）
* 必須取得データ: 本数字、ボーナス、**キャリーオーバー金額（1円単位）**

### **STEP 3: 解析と可視化（Analysis & Viz）**
* スクリプト: `analyze_v4.py`
* **必須出力グラフ**:
    1.  **重力スコア・ヒートマップ**: 「なぜその数字か」が一目でわかる棒グラフ。
    2.  **波動構成チャート**: 選出理由（深さ・相性など）の内訳グラフ。
    3.  **損益シミュレーション**: 直近10回で戦った場合の勝敗グラフ。

### **STEP 4: 最終出力**
* グラフと併せて、**「ファイナル・アンサー（1口）」**のみを出力する。
